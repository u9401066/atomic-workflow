# Decision Log

| Date | Decision | Rationale |
|------|----------|-----------|
| 2026-03-06 | Atomic Workflow spec adopts a baseline-plus-variant-overlay data model with separate parsers for baseline phase files and variant overlay files. | The existing markdown corpus uses different heading semantics in baseline and variant documents, including range headings and legacy variant-only IDs. A single parser contract was inconsistent with the real data and would block reliable parsing, validation, and variant comparison. |
| 2026-03-06 | Atomic Workflow now targets a shared core engine with two primary clients: a VS Code authoring client and a Streamlit presentation/review client. | The system is intended for both engineering and non-engineering stakeholders. VS Code is the best authoring surface, while Streamlit is better for graph interaction and dashboard review. Both must share the same parser, resolver, validation, and graph contracts to avoid duplicated logic. |
| 2026-03-08 | 採用 React 19 + @xyflow/react (React Flow 12) 取代 mermaid.js 作為 VS Code extension webview 的互動式 node editor。同時引入 Vite (webview bundler) 和 Tailwind CSS (styling)。 | (1) 需求本身就是互動式 node-based editing，mermaid.js 只能 read-only rendering。(2) React Flow 是業界標準 — Dify、Flowise、LangFlow 3/5 大 visual workflow 工具都使用。(3) React 生態成熟、文檔完善、社群活躍（35.5k⭐）。(4) 支援 custom nodes、bidirectional editing、drag-drop、zoom/pan。 |
| 2026-03-08 | 採用 Step-level YAML Block 作為步驟級別的結構化屬性儲存格式（§4.5），形成三層資料模型（§4.6）。 | (1) 純 regex 抽取角色/裝備/警告既脆弱又有資訊損失。(2) Bidirectional editing 需要穩健的寫回機制，ruamel.yaml roundtrip 可確保格式安全。(3) 新屬性（duration_sec, condition）只需加 YAML key，不用改 parser。(4) v0 相容：無 YAML block 時 fallback 到 regex。 |
| 2026-03-08 | MCP 寫入操作採用嚴格的 8 階段 Write Hook Pipeline（§11.5.5），pre-write 4 hooks + post-write 4 hooks，任一失敗即 reject 或 rollback。 | 防止 MCP/agent 的操作錯誤破壞工作流檔案。Pre-write 攻防：schema、immutable fields、structural integrity。Post-write 攻防：re-parse、re-validate、YAML-body sync。全程 audit trail 記錄至 .atomic-workflow/audit.jsonl。 |
| 2026-03-08 | Cherry-pick BPMN/FHIR/DMN/CMMN 概念寫入 SPEC.md，不採用完整 BPMN XML。引入 7 個 node_type（task/decision/parallel_start/parallel_end/subprocess/event/milestone）+ 7 個 refs type（triggers/depends_on/parallel_with/uses_output_of/shares_resource/escalates_to/compensates）。 | (1) BPMN XML 違反 Markdown-first 原則、AI 不友善。(2) Cherry-pick 概念詞彙保留表達力（BPMN Task→task, Gateway→decision/parallel, Sub-Process→subprocess, Event→event）。(3) refs 借鏡 FHIR relatedAction 的 relationship enum，但用自然語言 condition。(4) shared/ 目錄實現 SQL 3NF 正規化，消除「物資補充」等多觸發共用流程的重複。 |
| 2026-03-06 | atomic-workflow 第一批實作切片建議採 baseline-only read path，先建立 domain models、baseline parser、repository、service facade 與最小 hook framework，再以 pytest 單元/整合測試固定契約。 | 目前 repo 尚無 src/ 與 tests，且現有語料缺少 frontmatter，因此最小有價值切片應優先實作 SPEC 的 v0 compatibility baseline parsing、DDD 邊界與可重用 hook utility，避免過早投入 variant/write path 導致重寫。 |
| 2026-03-06 | 專案測試入口統一為 `uv run test` / `uv run pytest`，不使用 `uv test`。同時在 `pyproject.toml` 以 `[project.scripts] test = "pytest:console_main"` 綁定測試腳本。 | `uv test` 不是專案命令，容易被誤解成外部套件解析，造成不必要的 resolver 記憶體消耗甚至 OOM。將 `test` 註冊為 project script 後，開發者與 hook/CI 都可以用一致入口執行 pytest。 |
